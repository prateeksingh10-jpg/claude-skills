### ⬆ NEW — Warning / Callout Box Pattern

When a slide requires a callout box to flag a caveat, limitation, or legal boundary (not a RAG status indicator), use the following three-token amber pattern consistently. This pattern was validated in production across Track 1 Leadership deck, April 2026.

| Token     | Hex       | Role                                      |
|-----------|-----------|-------------------------------------------|
| Background| `#FFF8E8` | Very light amber — box fill               |
| Border    | `#ED8B00` | Orange (functional) — 1.0–1.5pt outline   |
| Text      | `#7A4000` | Dark amber — readable on `#FFF8E8` at 9–10pt |

```python
# Standard amber callout box (python-pptx)
AMBER_BG     = RGBColor(0xFF, 0xF8, 0xE8)
AMBER_BORDER = RGBColor(0xED, 0x8B, 0x00)
AMBER_TEXT   = RGBColor(0x7A, 0x40, 0x00)

rect(slide, x, y, w, h, AMBER_BG, line=AMBER_BORDER, lw=1.0)
tb(slide, x+0.15, y+0.07, w-0.30, h-0.12,
   "⚠  Callout text here.", sz=10, color=AMBER_TEXT, wrap=True)
```

**When to use:** Honest gaps (e.g. SAS not supported), legal or consent requirements, peer evidence caveats, anything that needs to be visible but not alarming. **Do not use** for RAG status cells in tables — use the on-palette severity gradient from `consulting-patterns.md` there.

## 3. Typography

> **Standard Deloitte materials:** **Calibri** (and Calibri Light) is the mandatory typeface for manual PowerPoint authoring.
>
> **Cloud Practice materials:** **Open Sans** (and Open Sans Light) replaces Calibri as the typeface for all Cloud-branded sales collateral, placemats, case studies, and PPT templates. Download Open Sans from Brandspace.

### ⬆ NEW — Font for Programmatic Generation (python-pptx via deloitte-pptx skill)

> **When building decks programmatically** using the `deloitte-pptx` skill and `python-pptx`, use **Aptos** throughout — not Calibri or Open Sans.
>
> The official 16:9 onscreen template (`assets/template.pptx`) bundled with the skill is **Aptos-native**. Its theme fonts, placeholder styles, and slide master are all set to Aptos. Specifying Calibri or Open Sans in code will cause font substitution warnings and visual inconsistency against the template's inherited styles.
>
> **Rule of thumb:**
> - Manual authoring in PowerPoint → Calibri (standard) or Open Sans (Cloud)
> - Programmatic generation via deloitte-pptx skill → **Aptos**
>
> This distinction does not affect brand compliance — Aptos is a Microsoft system font in the same humanist sans-serif family and renders correctly on all Deloitte-standard Windows machines.

```python
# Always use this in python-pptx builds
FONT = 'Aptos'
run.font.name = FONT
```

## 4. Slide Layout Templates

### 4.1 Cover / Title Slide
- **Backgrounds available:** White, Black
- **Logo:** Top-left corner
- **Headline:** Bottom-left, Calibri Light (or Open Sans Light for Cloud), Green, up to 3 lines
- **Subtitle:** Bold, small, below headline
- **Image:** Circular crop or full-bleed; centered or right-aligned; send to back
- **Badge:** "Making an Impact That Matters" – bottom-right
- **Optional note box:** Top-right, teal border, with image replacement instructions (remove in final)

### 4.2 Standard Content Slide
- **Page Title:** Top-left, Light weight font, black
- **Page Subtitle:** Below title, Cool Gray
- **Body Area:** Full width or split (text left / image right)
- **Footer:** Left – copyright; Right – presentation title + page number
- **Text levels:** 5 predefined levels (see Typography section)

### 4.3 Two-Column Text Slide
- Left column and right column, equal width
- Each column uses the same 5-level text hierarchy

### 4.4 Text + Image Slide
- Left: Text with title, body, and bullets
- Right: Full-height or partial image (rectangular or circular)

### 4.5 Section Divider / Title Break Slide
- **Option A:** Bold title (black) + subtitle (Deloitte Green) over a full-bleed image
- **Option B:** Same text layout over a white or toned background
- No footer or page number on divider slides

### 4.6 Key Statement / Quote Slide
- Large-format single statement, centered or left-aligned
- Highlight key phrases in: *italic*, brand color, or a contrasting hue
- Backgrounds: White, Black, Deloitte Green, Dark Green, Teal, Dark Blue

### 4.7 Contents / Agenda Slide
- Simple list of items with page numbers (right-aligned X placeholders)
- Optional: right-side image panel
- Sub-items with bullet prefix (•)

### 4.8 Agenda Table Slide
- 4-column table: Topic | Content | Person Responsible | Timing

## 6. Image Guidelines

### 6.1 Primary Images (Cloud Visual Identity)

Primary images embody possibility through light, color, and data-inspired abstraction. Used on title slides, email headers, and document covers (circular crop).

- **Never** use an image of actual clouds for the circular primary image.
- Use abstract images focused on: amplification, disruption, innovation, transformation, and possibility.
- Use images with **black or transparent backgrounds** to accentuate light, incandescence, and motion.
- *Exception:* White backgrounds and suitable imagery may be substituted for Cloud Alliance-specific events or materials.

### 6.2 Secondary Images (Cloud Visual Identity)

Secondary images are typically used on divider slides. Two types are acceptable:

**Abstract / data-inspired:**
- Draw from the same visual world as primary images; colors should match Deloitte's palette.
- Apply the same three rules as primary images (no literal clouds, abstract focus, dark backgrounds preferred).

**Photography:**
- Show unique perspectives on industry or the human side of cloud adoption.
- Images should convey: discovery, change, authenticity, and energy.
- When depicting industry use cases, prioritize images centering **human connection and interaction**.
- Panoramic night images from nature may be used if they contain dynamic light and motion echoing the primary imagery.
- Images must feel modern and dynamic — buyers, sectors, and/or technology should be central.
- **Avoid** overly stock or posed photography; choose candid and spontaneous imagery instead.
- **Do not** use literal cloud imagery in photography.

### 6.3 Standard Image Guidelines (all materials)

| Style | Description |
|-------|-------------|
| Full bleed | People or object, edge-to-edge |
| Full bleed with negative space | Subject offset to allow text overlay |
| Cut-out style | People on black or white backgrounds |
| Circular crop | Image masked in a circle shape |
| Monochrome / B&W | Mood imagery |
| Green accent | Naturally occurring or styled green within image |
| Solid color backgrounds | Green or teal fills |
| Concentric / portal imagery | Tunnel perspective; people interacting with a circle |

### 6.4 Image Placement Rules
- Cover images: use circular mask or full-bleed; always **Send to Back** after insertion.
- To replace: right-click → Change Picture, or delete and use Insert > Picture.
- To resize without distortion: use Picture Tools > Format > Crop; **hold Shift** when resizing from corners.
- Secondary images: right-click → Change Picture; crop to fit.
- Don't use overly posed or stock photography for Cloud materials

### ⬆ NEW — Boilerplate Stripping in Programmatic Builds

The Deloitte template contains several boilerplate text strings in footer placeholders that must be removed before delivery. When building with python-pptx, a naive text-search strip risks accidentally clearing content slides if those phrases happen to appear in slide body copy.

**Correct approach: strip by y-position, not by content alone.**

Only strip shapes where `shape.top / 914400 > 6.0` (i.e. shapes positioned below the 6-inch mark — the footer zone). This confines the strip to footer placeholders and leaves all content-area shapes untouched.

```python
def strip_boilerplate(prs):
    """Remove template boilerplate from footer zone only (y > 6.0 inches)."""
    bad_strings = [
        'member firms', 'dttl', 'insert appropriate copyright',
        'to edit, click view', 'presentation title',
        'click view > slide master', 'slide master'
    ]
    for slide in prs.slides:
        for shp in slide.shapes:
            if not shp.has_text_frame:
                continue
            try:
                y_inches = shp.top / 914400   # EMU → inches
                if y_inches < 6.0:
                    continue                   # Never touch content-zone shapes
            except:
                continue
            text_lower = shp.text_frame.text.lower()
            if any(b in text_lower for b in bad_strings):
                for para in shp.text_frame.paragraphs:
                    for run in para.runs:
                        run.text = ''
```

**Always call `strip_boilerplate(prs)` before `prs.save()`.**

Known boilerplate strings in the standard template:
- `"Member firms and DTTL: Insert appropriate copyright"`
- `"[To edit, click View > Slide Master > Slide Master]"`
- `"Presentation title"` (bottom-right footer)

## 8. Bullet & Text Hierarchy

```
Level 1  Standard body text — no bullet
• Level 2  First level bullet (filled circle)
  − Level 3  Second level bullet (en dash)
    ◦ Level 4  Third level bullet (open circle)
      − Level 5  Fourth level bullet (en dash, further indent)
```

- 6pt line space between Level 1 (and Level 2) items
- No additional line space between Level 3–5 sub-bullets
- Use **Increase/Decrease List Level** buttons in Home > Paragraph to navigate levels

## 9. Chart & Data Visualization Standards

- **Font:** Calibri (standard) or Open Sans (Cloud) in Cool Gray 9 for axis labels, data labels, and legends
- **Chart headings:** Bold, Black, 13/18pt
- **Source & Note lines:** 9pt, Cool Gray 9–10, below chart
- **Highlight series:** Use Deloitte Green or Accessible Blue to call out one key data series; all others in Cool Gray tones
- **Color assignment for multi-series charts:** Draw from the Blue, Teal, and Green palettes in order; never use Red/Orange/Yellow except as functional indicators
- **Grid lines:** Light Cool Gray 2 or Cool Gray 4; thin weight
- **No 3D effects** on any chart type
- **Legends:** Positioned below or to the right of the chart; 9–11pt

## 10. Table Formatting Standards

- **Column headers:** Bold, white text on Deloitte Green or Dark background
- **Row titles:** Bold, Black
- **Body cells:** Black or Cool Gray 11, 13/18pt
- **Alternating rows:** White and Cool Gray 2 (`#D0D0CE`) for readability
- **Subtotal / Total rows:** Slightly darker background (Cool Gray 4) or bold border
- **Verdana** is permitted for column headings in tables (as noted in the source guide)
- Borders: thin Cool Gray lines; no heavy borders except header row separator

## 14. Icon Guidelines

### 14.1 Icon Styling Rules

- **Style:** Line/outline icons preferred; flat, single-weight strokes. No skeuomorphic or 3D effects.
- **Stroke weight:** Consistent across all icons in a composition — typically 1.5–2pt at standard slide size. Never mix thin and thick stroke icons on the same slide.
- **Fill:** Icons are generally unfilled (outline only). Filled variants may be used for emphasis or active states — use sparingly.
- **Corner style:** Rounded corners and rounded line caps preferred.
- **Size:** Standardize icon size within any single slide. Common sizes:
  - Small (inline with text): 16–20pt
  - Medium (standalone label+icon): 32–48pt
  - Large (hero/feature icon): 64–96pt
- **Color:**
  - Default: Black or Cool Gray 9 (`#75787B`)
  - Accent: Deloitte Green (`#86BC25`), Accessible Green (`#26890D`), Accessible Blue (`#007CB0`), or Accessible Teal (`#0D8390`)
  - On dark backgrounds: White or Deloitte Green
  - Never use Bright Green, Bright Teal, or Bright Blue for icons
  - Functional icons (status indicators): Red (`#DA291C`), Orange (`#ED8B00`), Yellow (`#FFCD00`)
- **Backgrounds:** Icons may sit inside a circle, square, or rounded-square container. Container fill should use a light palette color (e.g., Teal 1, Blue 1, Green 1) with the icon in the corresponding accessible dark color.
- **Alignment:** Always align icons to a consistent grid. In multi-icon layouts, use equal spacing between icon+label units.
- **Source:** Always download from the official Deloitte Brand Space icon library.

### 16.3 Card Layout Pattern

> **Rule:** All content cards use white fill and a gray border. **No coloured top bars. No shadows.** Card body text must never overflow — enforce padding, word wrap, and a font floor.

#### Card Anatomy

```

│  Card Title  (Bold, 12pt, navy) │
│                                 │
│  Body text — Open Sans 10pt min │

│                                 │
└─────────────────────────────────┘

```

#### Token Reference

| Token | Value | Usage |
|-------|-------|-------|
| Card fill | `#FFFFFF` | All card bodies — never use colored fills |
| Card border | `#D0D0CE` Cool Gray 2, 0.75pt | All four sides |
| Card title | `#012169` Blue 6, Open Sans Bold 12pt | Always navy |
| Card body | `#1A1A1A` near-black, Open Sans 10pt min | Floor is 10pt — shrink text, not box |
| Shadow | None | Never apply drop shadows or glow effects to any shape |
| Teal top bar | Not used | Removed — do not add coloured accent bars to cards |

> **Removed tokens:** `Top bar — primary (#0097A9)`, `Top bar — secondary (#046A38)`, `Callout box fill (#E8F7F9)`, `Callout box border (#0097A9)` — these were present in earlier builds and must not be carried forward.

#### Text-in-Box Enforcement Rules

1. **Padding:** Set `margin_left = margin_right = margin_top = margin_bottom = Pt(8)` on every text frame inside a card.
2. **Word wrap:** `text_frame.word_wrap = True` — always.
3. **Auto size:** `text_frame.auto_size = None` — do NOT auto-grow boxes; fit text to the fixed box.
4. **Font floor:** Never set body text below 10pt. If content cannot fit at 10pt, shorten the copy — do not shrink the font further.
5. **Overflow check:** After placing text, verify that `text_frame` content does not visually exceed the shape bounds. If it does, reduce content before reducing font size.

```python
from pptx.util import Pt
from pptx.enum.text import PP_ALIGN

def set_card_textframe(tf, body_font_size=Pt(11)):
    tf.word_wrap = True
    tf.auto_size = None
    tf.margin_left   = Pt(8)
    tf.margin_right  = Pt(8)
    tf.margin_top    = Pt(8)
    tf.margin_bottom = Pt(8)
    for para in tf.paragraphs:
        for run in para.runs:
            if run.font.size and run.font.size < Pt(10):
                run.font.size = Pt(10)   # enforce floor
```

### 16.5 Presentation Delivery Mode

> **When to use:** Any deck that will be presented live in a room or on a call — not handed to someone to read. This covers all CoCo Adoption Programme session decks. Applies on top of Pattern B (Section 16.4).

The audience will not read the slide. They are listening to the presenter. The slide is a visual anchor for what is being said, not a document. Every design decision should serve that reality.

---

#### The Core Rule

> **The slide title IS the message. Everything else is support.**

A presenter should be able to walk through the deck showing only titles and tell the complete story. If removing the body text breaks the narrative, the title needs to be rewritten — not the body kept.

---

#### Font and Size Rules

| Element | Minimum size | Notes |
|---------|-------------|-------|
| Slide title | **22–24pt bold** | Bold; must fit on one line; reduce size before wrapping |
| On-screen body text | **20pt** | Hard floor. Nothing smaller appears on the slide. |
| Item labels / card titles | **18pt** | Absolute minimum for any visible label |
| Speaker notes | Any size | Notes are for the presenter only — no size restriction |
| Footnote / attribution | 9pt | Only exception — truly secondary, non-content text |

**Why 20pt:** Text below 20pt loses impact on a projected screen. 18pt is the absolute floor; 20pt is the standard. If the content only works at 10–12pt, it belongs in speaker notes, not on the slide.

---

#### Density Rules

| Rule | Limit |
|------|-------|
| Ideas per slide | **1** — the title states it, the body supports it |
| Bullet points / items per slide | **3–4 max** |
| Words per bullet | **6–8 max** — a label, not a sentence |
| Tables on a live slide | **Avoid** — use a visual summary instead; put the full table in notes or appendix |
| Anatomy / reference content | **Never on a live slide** — always notes or appendix |

---

#### Speaker Notes — Mandatory

Every slide that was trimmed for on-screen density must have speaker notes containing:
- The full detail that was removed from the slide
- The talking points the presenter should hit
- Any examples, caveats, or nuances cut from the visual

In python-pptx, notes are added via:

```python
def add_notes(slide, text):
    notes_slide = slide.notes_slide
    tf = notes_slide.notes_text_frame
    tf.text = text
```

**Rule:** If a bullet was cut from the slide, its content must appear in notes. Nothing is lost — it moves to the presenter's script.

│  TITLE — the one thing the audience takes away       │  22–24pt bold black (must fit one line)

│                                                      │

## ⬆ NEW — Mandatory Deck Construction Rules (April 2026)

These rules apply to ALL decks produced for the CoCo Adoption Programme. They override any conflicting guidance elsewhere in this document.

### Logo & Branding
- **No "Together makes progress" tagline.** The Deloitte template has a "tagline logo lockup" variant — do NOT use it. Use the plain Deloitte wordmark layouts (odd-numbered title layouts: 1, 3, 5, 7, 9, 11). The tagline does not exist in our context.
- **No Cloud sub-brand lockup** unless explicitly producing Cloud-branded materials.

### Slide Headers
- **Slide titles must be prominent.** Minimum 28pt for content slide titles. The title is the first thing the audience reads — it must be visually dominant.
- When using freeform (Pattern B) slides, create the title textbox at 28–32pt bold, not the default placeholder size.

### Footers
- **No footers.** Do not include page numbers, copyright lines, "Presentation title," or "Member firms and DTTL" text on content slides. The only exception is the final end slide which carries the copyright and Deloitte mark.
- When using the template, actively suppress or clear all footer placeholders and master-inherited footer text.

### Content Boxes
- **Use rectangular boxes (not rounded rectangles).** All content containers — cards, tiles, callout boxes — must be rectangular.
- **Text must fit within boxes.** If text overflows a box, either reduce the text or increase the box size. Never allow text to clip or overflow. Test at the intended font size before finalising.

### Section Dividers
- **Do not insert section divider slides** unless the presenter explicitly requests them. Content flows directly from slide to slide without divider interruptions.

### Agenda / Contents Slide
- **Every deck MUST include an Agenda or Contents slide** immediately after the title slide (slide 2 position). This slide lists all content slide titles with their corresponding page numbers. It serves as both the session roadmap and a reference for the audience.

## ⬆ NEW — 17. Content Standards — Learned from CoCo Practitioner Deck Feedback (April 2026)

These rules govern the *what goes on a slide* layer — distinct from the visual/brand rules above. They apply to all practitioner, teaching, and enablement decks and were validated against explicit user feedback on the CoCo Master Deck build.

---

### 17.1 No Org Attribution on Content Slides

> **Rule:** Programme names, org names, team names, and session dates must not appear on individual content slides.

Acceptable locations: cover slide, close/Q&A slide only.

**Remove from content slides:**
- "Snowflake CoE" / "Deloitte USI Snowflake CoE"
- "CoCo Adoption Programme"
- "April 2026" (or any date stamp)
- Any footer line combining org + programme + date

**Why:** Attribution on content slides fragments audience attention, looks bureaucratic, and limits the deck's reuse life.

---

### 17.2 Two-Level Header Rule

> **Rule:** Every content slide must have both a main title and a sub-header.

| Element | Spec | Purpose |
|---------|------|---------|
| Main title | Bold, one line, ≥ 24pt (Delivery Mode ≥ 22pt) | States the single key message |
| Sub-header | Teal `#0097A9`, 13pt bold, italic, y ≈ 0.9" | Tells the audience *what is being compared, explained, or shown* |

**Sub-header examples:**
- "Comparing Claude.ai to Snowflake Cortex Copilot"
- "How Copilot fits into a Snowflake developer's daily workflow"
- "What CoCo does well — and where to apply judgment"
- "A decision framework for choosing the right AI entry point"

If the audience cannot tell from the title alone what the slide is *about*, a sub-header is mandatory.

---

### 17.3 Neutral and Defensible Comparative Claims

> **Rule:** Never assert a competitor product weakness by name. All comparative claims must be true for any general-purpose AI tool.

| Instead of… | Write… |
|-------------|--------|
| "ChatGPT generates generic SQL" | "General-purpose AI produces ANSI SQL with no knowledge of Snowflake-specific syntax, functions, or patterns." |
| "GPT-4 doesn't understand Snowflake context" | "General-purpose models have no visibility into your Snowflake environment, schema, or warehouse configuration." |

**Test:** Could the same sentence be applied to *any* general-purpose LLM? If yes, it's safe. If it names a product, rewrite it.

---

### 17.4 Positive Framing for Limitation / Guardrail Slides

> **Rule:** When a slide covers tool limitations, frame it as a judgment guide — not a failure list.

**Title pattern:** Use "know before you go" framing, e.g.:
- "Where to apply your judgment — know before you go."
- "CoCo's current boundaries — and how to work within them."
- "Guardrails that make CoCo safe in production."

**Body pattern:** For each limitation, state the limitation and the practical response:
- What CoCo does in this case
- How the practitioner should handle it

**Visual pattern:** Use the amber callout box (Section 2, Warning/Callout Box Pattern) for each limitation item:
- Background: `#FFF8E8`
- Border: `#ED8B00`, 1pt
- Text: `#7A4000`

---

### 17.5 No Internal Artefact Names

> **Rule:** All references to internal documents, CoE skill file names, internal repositories, or internal guide titles must be removed before delivery.

**Remove:**
- Internal file names: `migration-standards`, `sas-analysis`, `cli-guide-v1.3`, any `*.md` CoE skill file
- Internal guide titles: "our CLI Guide", "the CoE playbook", "Track 1 materials"
- Internal repo paths or org GitHub handles

**Replace with:**

### 17.8 Takeaway Bar Standard (pptxgenjs)

> **Rule:** Every slide with a closing key-message line must use a solid coloured rectangle — bold white text, centre-aligned. Never a thin divider line with plain text below it.

```javascript
// Standard takeaway bar — h=0.45", full content width
function addClosingLine(slide, text, y) {
  slide.addShape(pres.shapes.RECTANGLE, {
    x: M.l, y: y, w: cw, h: 0.45,
    fill: { color: "0097A9" }, line: { color: "0097A9" },
  });
  slide.addText(text, {
    x: M.l + 0.15, y: y, w: cw - 0.3, h: 0.45,
    fontSize: 13, bold: true, fontFace: "Open Sans",
    color: "FFFFFF", align: "center", valign: "middle", margin: 0,
  });
}
```

**Y-position constraint:** bar height is 0.45". Place so bar ends ≤ 5.60" (slide height 5.625"). All content above must end before the bar starts — verify `y_content_end < y_bar`.

**Card/table size limits to ensure bar fits:**

| Content type | Max height |
|---|---|
| Two-panel cards (2 × full-height) | body panel ≤ 3.2" |
| Three-column prompt cards | body panel ≤ 3.0" |
| Tables with 5–6 rows | h ≤ 4.0", rowH ≤ 0.65" |
| Resource grid (2×2) | cell height ≤ 1.9" |
